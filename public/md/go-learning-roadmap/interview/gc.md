# 垃圾回收 (GC)

## 概述

Go 使用 **并发标记-清除 (Concurrent Mark-Sweep)** 算法进行垃圾回收。

## GC 算法演进

| 版本 | 特点 |
|------|------|
| Go 1.3 | STW (Stop The World) 标记，全部暂停 |
| Go 1.5 | 三色标记法，并发标记 |
| Go 1.6 | 减少 STW 时间到 < 1ms |
| Go 1.8 | 混合写屏障 (Hybrid Write Barrier) |
| Go 1.19+ | 引入 GOMEMLIMIT (内存限制) |

## 三色标记法

```
白色: 未被检查的对象（可能被回收）
灰色: 已被引用但引用的对象未检查完
黑色: 已检查完毕（存活对象）
```

### 标记过程

```
初始: 所有对象为白色
      ↓
从根对象开始扫描 → 标记为灰色
      ↓
扫描灰色对象的引用 → 将引用的白色对象标记为灰色
      ↓
当前灰色对象标记为黑色
      ↓
重复直到没有灰色对象
      ↓
剩余的白色对象 = 可回收垃圾
```

### 写屏障 (Write Barrier)

防止并发标记过程中指针修改导致的问题：

```go
// 混合写屏障 (Go 1.8+)
// 1. 写入时，将新指针标记为灰色
// 2. 写入时，将旧指针也标记为灰色
```

## GC 触发条件

1. **主动触发** - 分配的堆内存达到阈值（默认为上次 GC 后堆内存的 2x）
2. **被动触发** - 定时触发（默认每 2 分钟至少一次）
3. **手动触发** - `runtime.GC()`

### GOGC 参数

```bash
# 默认值 100 表示: 当当前堆内存达到上次 GC 后堆内存的 200% 时触发 GC
GOGC=100 ./myapp

# 关闭 GC (仅用于测试)
GOGC=off ./myapp

# 更频繁的 GC (更低的延迟，更高的 CPU 开销)
GOGC=50 ./myapp
```

### GOMEMLIMIT (Go 1.19+)

```bash
# 设置软内存限制
GOMEMLIMIT=512MiB ./myapp
# 当堆内存接近此限制时，GC 会更积极地运行
```

## GC 调优建议

1. **减少分配** - 最有效的优化方式
2. **使用 sync.Pool** - 复用对象减少 GC 压力
3. **合理设置 GOGC** - 低延迟场景可降低 GOGC
4. **使用 GOMEMLIMIT** - 容器化部署时推荐设置
5. **避免 Finalizer** - 会阻止对象回收

## 查看 GC 信息

```bash
# GC 日志
GODEBUG=gctrace=1 ./myapp

# 输出示例:
# gc 1 @0.008s 3%: 0.039+1.9+0.083 ms clock, 0.16+7.6/1.9/0.33 ms cpu, 4->4->1 MB, 4 MB goal, 12 P
```
